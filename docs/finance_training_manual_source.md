# 歲悅 Finance OS 會計系統教育訓練手冊

版本：v1.0  
適用對象：一般員工、主管、行政部門主任、總務、人資、會計、執行長  
建議訓練時間：60 至 90 分鐘  
上線建議：先由 2 至 3 位種子人員試營運，再逐步開放全員使用。

## 1. 系統目標

Finance OS 是歲悅內部會計申請與簽核系統，用來集中處理支出申請、發票開立、收款確認、付款入帳、傳票查詢與三表資料。系統設計原則是「易使用上手、資料完整、即時同步、避免重複計算」。

使用者不需要理解完整會計分錄才能送單，但必須正確填寫申請內容、公司法人、部門、金額與附件。會計會在最後關卡確認憑據與金流，系統才會正式產生傳票並寫入帳冊。

## 2. 角色與權限

| 角色 | 主要任務 | 常用功能 |
|---|---|---|
| 一般員工 | 建立申請、上傳憑據、查看自己的申請與待補件事項 | 新增申請、放款申請、簽核管理、通知中心 |
| 課長 / 部門主管 | 檢查費用合理性、部門預算與附件完整性 | 簽核管理、我的簽核、歷史紀錄 |
| 行政部門主任 | 檢查行政流程、付款或發票資料、系統設定異動 | 簽核管理、系統設定、通知中心 |
| 總務 | 採購完成後補上實際支出與發票憑據 | 採購流程、簽核管理 |
| 人資 | 人事費用相關審核 | 人事費用申請、簽核管理 |
| 會計 | 確認憑據、付款資料、傳票與三表入帳 | 傳票查詢、三表總覽、收款情形、會計科目 |
| 執行長 | 最終授權、重大費用核准、系統設定檢核 | 簽核管理、儀表板、系統設定 |

## 3. 登入與會計工作台

1. 開啟公司提供的 Finance OS 網址。
2. 使用公司帳號登入，測試階段可使用內建測試帳號。
3. 登入後會直接進入會計工作台。
4. 依左側功能選單進入儀表板、申請、簽核、傳票、報表或系統設定。
5. 若畫面沒有正確顯示權限，請先登出再重新登入。

注意：正式上線後，建議使用公司正式 domain，例如 `finance.suiyuecare.com`，並確認 Google OAuth 與 Supabase Auth 都已加入正式網址。

## 4. 儀表板

系統依角色顯示不同儀表板：

- 員工儀表板：只看最近申請與最新通知。
- 會計儀表板：顯示公司收入、支出、淨利、部門收入分布、部門支出分布與真實三表資料。

所有財務數字原則上都以系統內真實資料為準，不補估、不造假。若沒有資料，系統應顯示空狀態或 0，而不是假數字。

## 5. 申請類型

新增申請時，先選擇申請類型，再填寫對應表單。

| 類型 | 使用情境 | 重點附件 |
|---|---|---|
| 費用報銷 | 員工已代墊費用，向公司申請報銷 | 發票、收據、憑據照片 |
| 付款申請 | 公司直接付款給廠商或外部單位 | 請款單、合約、發票 |
| 預支申請 | 員工先向公司預支款項，之後核銷 | 預支申請書、預計用途 |
| 零用金申請 | 部門申請零用金補足或初次申請 | 一般申請需憑據；初次申請免憑據 |
| 差旅申請 | 出差前申請交通、住宿、膳雜等費用 | 住宿、交通、停車或其他憑據 |
| 採購申請 | 物品、設備、服務、軟體或保險採購 | 採購明細、報價、後續發票 |
| 退費申請 | 退還客戶、學員、案家或合作單位款項 | 退費明細、原發票或退款證明 |
| 人事費用申請 | 薪資、獎金、勞報、保費、資遣等 | 薪資表、人事費用表、勞務報酬單 |

## 6. 一般申請操作流程

1. 點選左側「新增申請」。
2. 選擇申請類型。
3. 填寫支出法人、支出部門、申請人、表單填寫日期。
4. 填寫申請內容與匯款資訊。
5. 上傳發票、憑據、存摺封面或相關文件。
6. 檢查即時 Excel 明細表與合計金額。
7. 可先按「暫存」；確認後按「送出申請」。
8. 到「簽核管理」確認是否已出現在「我的簽核」或「歷史紀錄」。

支出法人會影響公司三表，支出部門會影響部門分類帳，請務必填對。申請人資料只是送件者身分，不等於費用歸屬。

## 7. 暫存表單

每張表單都可以暫存。暫存表單只會顯示給建立者本人，其他人不應看到。

操作方式：

1. 在表單填寫途中按「暫存」或「儲存草稿」。
2. 回到「放款申請」或「簽核管理」中的「暫存表單」。
3. 點選「繼續填寫」可回到表單。
4. 若不再使用，可點選刪除。

注意：瀏覽器基於安全限制，部分本機檔案可能無法在草稿中完整恢復。正式送出前，請再次確認附件是否仍在清單中。

## 8. 懶人費用申請

懶人費用申請藏在「支出申請」中，適用於費用報銷、付款申請、預支申請與零用金一般申請。

使用方式：

1. 進入「放款申請」後點選懶人申請。
2. 上傳一張或多張發票或憑據照片。
3. 按 OpenAI 解析憑據。
4. 系統會讀取發票號碼、日期、品項、數量、單價、未稅、營業稅與含稅金額。
5. 右側即時 Excel 明細表可直接修改。
6. 送出後，系統會將明細表轉成可下載 Excel 附件，憑據照片會彙整成 PDF 給主管檢核。

如果辨識結果不準，申請人仍須手動修正。送出前，申請人對資料正確性負責。

## 9. 人事費用申請

人事費用申請適用整批薪資、獎金、勞務報酬、勞健保、勞退、補充保費與資遣費。

人事費用項目：

1. 50薪資
2. 獎金
3. 9A/9B勞報單
4. 勞保
5. 健保
6. 勞退
7. 二代健保補充保費
8. 資遣費
9. 其他

填寫重點：

- 申請人名字、申請人部門、申請人法人會由帳號預設帶入。
- 支出法人與支出部門是三表與部門分類帳認列依據。
- 人事費用明細表欄位包含員工編號、姓名、類別、薪資月份、匯款金額、銀行、分行、帳號與備註。
- 除員工編號與備註外，其餘欄位為必填。
- 9A/9B 勞務報酬單必須上傳勞務報酬單。
- 填寫即時 Excel 明細表時，按 Enter 可新增下一列並跳到同一欄。
- 主管簽核時會看到可下載的 Excel 明細檔與彙整後的附件。

人事費用會依項目對應不同科目，例如薪資、獎金、勞保、健保、勞退、補充保費、資遣費與勞務費，以利三表正確呈現。

## 10. 差旅申請

差旅申請不需要先選第一關費用性質，進入後直接填寫：

- 差旅類型
- 關聯專案或活動
- 出差事由
- 預期成果
- 出發日期與時間
- 返回日期與時間
- 出發地與目的地
- 是否申請住宿費、交通費、膳雜費、停車費或其他費用

若申請住宿、交通、停車或其他費用，請上傳憑據並填寫金額。膳雜費可依公司差旅管理辦法直接帶入金額。系統會把差旅項目帶入即時 Excel 明細表。

## 11. 採購申請

採購申請使用於物品、設備、服務、軟體、工程與保險採購。

流程重點：

1. 申請人先填預估採購內容與預估金額。
2. 簽核核准後，進入採購流程。
3. 申請人確認收到採購商品。
4. 總務採購完成後，上傳發票憑據並填寫最終實際支出。
5. 會計以最終實際支出入帳。

採購申請一開始的金額只是預估，最後三表認列應以總務上傳的實際憑據數字為準。

## 12. 發票開立與收款

發票開立也需要走簽核流程。發票流程完成後，收入會列入帳上收入；但現金流量表只有在會計確認已收款並經行政部門主任複核後才會認列現金流入。

收款情形分為：

- 單張發票
- 整批發票

會計確認收款時，應上傳收款證明，例如入帳截圖、銀行明細或其他收款文件。行政部門主任複核通過後，現金流量表才更新。

## 13. 簽核流程

一般支出申請的八階段簽核如下：

1. 申請人送件：確認申請內容、附件與 Excel 明細。
2. 申請人上一層級主管：確認業務必要性與金額合理性。
3. 申請人部門主管：確認部門預算與成本歸屬，若與上一層主管相同則跳過。
4. 行政部門主任：確認行政流程、付款資料與附件完整性。
5. 會計：確認會計科目、稅務處理、憑據與匯款資料。
6. 執行長：最終授權。
7. 申請人確認：確認已撥款或已取得會計開立的發票。
8. 會計確認入帳：產生傳票並寫入分類帳與三表。

每一關都可以填寫備註、上傳附件、核准、退件或退回上一關。若需要加簽，該關卡通過後，加簽人會成為下一關。

## 14. 附件與下載

主管與會計在簽核時應可以下載：

- 即時 Excel 明細表產生的線下 Excel 檔
- 憑據照片彙整 PDF
- 存摺封面獨立附件
- 申請人、主管或會計在各簽核關卡上傳的補充文件

存摺封面不應與憑據照片合併，應獨立提供下載，避免主管檢視憑據時混淆。

## 15. 傳票與三表

三表包含資產負債表、損益表與現金流量表。系統資料原則：

- 公司支出的三表追溯「支出法人」。
- 部門分類帳追溯「支出部門」。
- 執行長核准後，現金流量表會反映實際付款金流。
- 最後一關會計確認入帳後，資產負債表與損益表才會更新。
- 系統不得在會計最後入帳時重複更新現金流量表。
- 發票一開立就認列收入；收款確認只影響現金流量表。

如果發現三表與申請單法人或部門不一致，應立即停止使用該筆資料，請會計與系統管理者檢查。

## 16. 系統設定

系統設定用來維護：

- 法人資訊
- 部門資訊
- 會計科目
- 每間公司的手續費金額
- 各層級可使用的功能

行政部門主任可啟動編輯，但設定異動需要執行長檢核後才可正式通過。平常未啟動編輯時，資料應保持唯讀，避免誤改。

## 17. 常見問題

### 送出後主管看不到待簽核

請先確認申請單是否已出現在「我的簽核」或「歷史紀錄」。若待簽核數字有增加但列表沒有內容，請重新整理頁面；若仍異常，請提供申請單號給系統管理者檢查簽核 steps。

### 暫存失敗

請確認目前是登入狀態，並重新整理後再試一次。若附件太大或瀏覽器阻擋本機檔案恢復，請重新上傳附件後再送出。

### OpenAI 辨識失敗

常見原因包含 API 額度不足、Edge Function 未部署、OPENAI_API_KEY 未設定、圖片太模糊或發票不完整。可先手動填寫明細，不要因 OCR 失敗而延誤送件。

### 附件下載不了

請確認檔案是否已上傳到 Supabase Storage，且 Storage RLS 與下載 URL 有正確設定。主管簽核時至少應能下載 Excel 明細與憑據彙整 PDF。

### 三表數字怪怪的

先檢查申請單的支出法人與支出部門，再檢查傳票科目。三表只應使用真實 ledger / voucher 資料，不應用假資料或補估。

## 18. 上線前驗收清單

| 檢查項目 | 合格標準 |
|---|---|
| 登入 | 可使用測試帳號或公司帳號登入 |
| 權限 | 不同角色看到的功能不同，且符合權限設定 |
| 暫存 | 暫存後只顯示給本人 |
| 送出申請 | 送出後出現在我的簽核與歷史紀錄 |
| 待我簽核 | 下一關人員看得到待辦並可開啟明細 |
| 附件 | Excel、PDF、存摺封面可下載 |
| 加簽 | 原關卡通過後，加簽人成為下一關 |
| 退回上一關 | 可填備註並回到上一關 |
| 傳票 | 會計最後確認後產生傳票 |
| 三表 | 法人、部門、收入、支出、現金流沒有重複計算 |
| OCR | 發票與存摺封面可辨識或可手動修正 |
| Supabase | 資料表、Storage、Edge Function、RLS 均正常 |

## 20. 上線後管理建議

- 第 1 週先以試營運方式上線，每日檢查申請、簽核、傳票與三表。
- 每週由會計檢查傳票與附件是否完整。
- 每月結帳前，先匯出或備份當月資料。
- 權限異動與系統設定異動需保留 audit log。
- API key、service role key、GitHub token 不得貼在前端或公開文件中。
- 員工若不確定申請類型，應先詢問會計或行政部門主任再送出。
